# Current Scope And Limits

v0.1.0 is the first public GitHub/source release of agent-fleet. It should be usable for local control-plane development and operator review, but it is not a hardened multi-user service.

## Supported Surfaces

v0.1.0 includes:

- Local Fastify API.
- React dashboard.
- `steward` CLI.
- Durable JSON-backed control-plane state.
- Steward Chat persistence.
- Decision review and corrections.
- Generic webhook connector boundary.
- Local command Worker adapter.
- SSH remote Worker adapter.
- Remote node readiness and workspace provisioning helpers.
- GitHub deploy-key lease helpers.
- Recovery endpoint and checkpoint records.
- Static docs site and GitHub Pages workflow.

## Operational Cautions

| Area | Limit |
| --- | --- |
| Storage | JSON state is inspectable and durable, but not a concurrent multi-user database. |
| Supervision | Worker lifecycle reconciliation exists, but production supervision loops still need hardening. |
| Remote fleet | Remote scheduling is selective and useful, but not a full cluster manager. |
| Connectors | Webhooks are generic gateway adapters, not provider-specific IM SDKs. |
| Authentication | Local development defaults bind to localhost. Internet-facing deployments need an authenticated gateway. |
| Merge flow | Review/merge cleanup is a Worker responsibility and must not be automated blindly. |
| Docs publishing | The workflow is compatible with GitHub Pages Actions, but release managers still own Pages settings and deployed URL review. |
| npm | The package remains `private: true`; public npm publishing is out of scope. Do not run `npm install agent-fleet`, because that npm package is a different project. |

## Security Constraints

Before exposing any endpoint outside localhost:

- Put the API behind authentication.
- Use connector allowlists and HMAC signatures.
- Keep Worker output out of connector replies by default.
- Avoid committing `.agent-fleet/`, `.env`, logs, local runtime state, or secrets.
- Use placeholders in docs and tests.
- Keep remote credentials owner-authorized and leased.

## Roadmap

Near-term roadmap:

- Multi-project status review.
- Better correction workflows.
- Double-check routing for high-impact decisions.
- Remote-first resource scheduling.
- Proxy-aware remote execution.
- More complete Worker resume/session management.
- Parallel Worker development through worktrees.
- Review/merge Worker flows with cleanup after verified merge.
- Stronger connector provider adapters.
- Database-backed control-plane state when JSON becomes limiting.

## Maintenance Checklist

Before claiming release or docs maintenance is complete:

- Run `npm run check`.
- Run `npm run build`.
- Run `npm run docs:build`.
- Confirm docs contain no private hosts, paths, IPs, tokens, logs, or local machine references.
- Confirm install docs point to source checkout, `npm link`, or `node dist/cli/main.js`, not the public npm package name.
- Trigger or observe the Pages workflow when docs deployment is part of the maintenance task.
- Do not publish an npm package or change package visibility until the owner explicitly decides to do so.
