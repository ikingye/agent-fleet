# Current Scope And Limits

v0.1.0 is the first public GitHub/source release of agent-fleet. It is useful for local control-plane development, operator review, and early Steward/Worker workflows, but it is not a hardened multi-user service.

The project remains source-installed from GitHub for now. Do not run `npm install agent-fleet`. The npm package name `agent-fleet` is already used by another project, and this repository keeps `"private": true` in `package.json` until the owner chooses a package name and publishing plan.

## Supported Surfaces

v0.1.0 currently supports:

- Local Fastify API for goals, Steward Chat, dashboard state, decisions, corrections, recovery, execution nodes, and webhook connectors.
- React dashboard for the compact owner-facing control plane.
- `steward` CLI for terminal status and chat after a source build, `npm link`, or direct `node dist/cli/main.js` usage.
- JSON-backed local state in `.agent-fleet/control-plane.json`, including durable `stewardMessages`.
- Steward decision review, correction records, checkpoints, Worker sessions, Worker reports, worktrees, remote nodes, events, and memory.
- Local command Worker adapter and SSH remote Worker adapter behind server-side boundaries.
- Remote readiness checks, git-ref workspace provisioning, deploy-key lease records, and recovery reporting.
- Generic webhook connector boundary for IM gateway experiments.

## Known Limits

| Area | Current limit |
| --- | --- |
| Storage | JSON-backed local state is durable and inspectable, but not a concurrent multi-user database. |
| Supervision | Worker lifecycle reconciliation and resume records exist, but production supervision loops still need hardening. |
| Remote scheduling | Remote offload is selective and useful for high-load work, but production scheduling is not a full cluster manager. |
| Remote state | Remote workspaces are disposable scratch checkouts. Git and local control-plane state remain the source of truth. |
| Connectors | Webhooks are generic gateway adapters, not a full WeChat/IM SDK or provider-specific integration. |
| Authentication | The default deployment is local-first and bound to localhost. Public exposure requires an authenticated gateway. |
| Merge flow | Review, merge, and cleanup are Worker responsibilities and must remain reviewable instead of being automated blindly. |
| npm | The package remains `private: true`; the public npm name is occupied by another project, so use the source checkout until the owner chooses a package name and publishing plan. |

## Operational Cautions

Run agent-fleet locally unless you have explicitly put it behind production controls:

- Keep the API bound to `127.0.0.1` by default.
- Put any public deployment behind an authenticated gateway.
- Use per-connector callback tokens, HMAC request signatures, and a sender allowlist.
- Keep Worker stdout/stderr, command lines, and resume mechanics out of connector replies by default.
- Do not commit `.agent-fleet/`, `.env`, logs, local runtime state, remote key copies, or secrets.
- Treat `workspacePath` as the project boundary for every owner goal.

Remote Workers are framework-heavy in v0.1.0. Before dispatching remote work, verify SSH, provider login or Worker runtime authentication, repository access through a GitHub deploy key, machine user, or GitHub App, proxy routing when needed, remote filesystem permissions, and scratch cleanup behavior.

## Roadmap Direction

The next maturity steps are:

- Multi-project status review.
- Better owner correction workflows.
- Double-check routing for high-impact Steward decisions.
- Remote-first resource scheduling with stronger capacity accounting.
- Proxy-aware remote execution and clearer fallback records.
- More complete Worker lifecycle, resume, and session reconciliation.
- Parallel Worker development through worktrees.
- Review/merge Worker flows with cleanup after verified merge.
- Stronger provider-specific connector adapters.
- Database-backed control-plane state when JSON becomes limiting.
